粘包
*现场
   发送abc def 接受abcdef
*原因
   应用层
      *接收方BtyeBuf设置过大(Netty默认1024)
   传输层-网络层
      * 滑动窗口：假设发送方256 bytes表示一个完整的报文，但是接收方处理不及时且窗口
大小足够大(大于256bytes)，这256 bytes字节就会缓冲在接收方的滑动窗口中，当缓冲窗口中
缓冲了多个报文就会粘包
      * nagle算法也会造成粘包

 半包
 *现象
   发送abcdef,接收abc def
 *原因
   *应用层
       接收方ByteBuf小于实际发送数据量
   *传输层-网络层
       *滑动窗口：假设接收方的窗口只剩下128bytes,发送方的报文大小是256bytes，
 这时候接收方窗口无法容纳发送发的全部报文，发送方只能先发送128bytes，等待ack后
 才能发送剩余部分，这就造成半包
  *数据链路层
        * MSS限制：当发送的数据超过MSS限制后，会将数据切分发送，就会造成半包

 本质
 发生粘包与半包现场的本质是因为TCP是流式协议，消息无边界
